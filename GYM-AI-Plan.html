1. Resumen Ejecutivo

GYM-AI es una aplicación web progresiva (PWA) para el seguimiento personalizado de entrenamientos físicos. Integra inteligencia artificial a través de la API de Groq para recomendar progresión de pesos y repeticiones basándose en el historial individual del usuario.

Este documento es una reescritura profesional de un proyecto existente (GYM 2.0), elevando la arquitectura desde una app de localStorage a un sistema full-stack con autenticación real, base de datos en la nube, sincronización multi-dispositivo, y backend seguro que protege las claves de API.

Objetivo principal: Construir un portfolio project de nivel profesional que demuestre competencias en Angular, Python/FastAPI, PostgreSQL, AWS y prácticas de desarrollo modernas (CI/CD, testing, arquitectura en capas).

3. Stack Tecnológico

🖥️ Frontend

Angular 21
Standalone components, signals, lazy routes. Conocimiento existente del equipo. Fuerte en enterprise.

⚙️ Backend

Python 3.11 + FastAPI
Liviano (128MB RAM), async nativo, OpenAPI automático, ideal para free tiers. Muy demandado en el mercado.

🗄️ Base de Datos

PostgreSQL (Supabase)
Free tier generoso (500MB). SQL universal para entrevistas. Supabase agrega Auth y real-time gratis.

🔐 Autenticación

Supabase Auth
JWT gestionado externamente. FastAPI valida el token. Sin implementar auth desde cero.

🤖 Inteligencia Artificial

Groq — llama-3.3-70b
Free tier generoso. Ultra rápido (inference en < 1s). Llamado solo desde el backend.

☁️ Hosting Backend

AWS EC2 t2.micro
Free 12 meses. FastAPI corre cómodo en 1GB RAM. AWS es el cloud más pedido en ofertas laborales.

🚀 Hosting Frontend

Vercel
Free tier permanente para Angular. Deploy automático por push. Preview por PR.

🔄 CI/CD

GitHub Actions
Integrado con GitHub. Free para repos públicos. Pipeline de test + deploy automático.

📦 Estado Frontend

NgRx Signals Store
Estándar Angular moderno. Signals reactivos. DevTools incluidos. Patrón predecible.

🧪 Testing

Pytest + Vitest
Pytest para FastAPI (estándar Python). Vitest para Angular (ya configurado, más rápido que Jest).

6. API REST — Endpoints Completos

Todos los endpoints requieren header Authorization: Bearer {jwt} salvo los de health check.

Auth / Health

GET/healthHealth check — no requiere auth
POST/auth/registerCrea perfil de usuario tras registro en Supabase

Días de entrenamiento

GET/daysLista todos los días del usuario, ordenados por position
POST/daysCrea nuevo día — body: { name, position }
PUT/days/{id}Edita nombre del día
DELETE/days/{id}Elimina día y todos sus ejercicios (CASCADE)
PUT/days/reorderActualiza positions — body: [{ id, position }]

Ejercicios

GET/days/{day_id}/exercisesLista ejercicios del día, ordenados por position
POST/days/{day_id}/exercisesAgrega ejercicio al día
PUT/exercises/{id}Edita ejercicio (nombre, sets, reps, descanso, unidad)
DELETE/exercises/{id}Elimina ejercicio
PUT/exercises/reorderReordena ejercicios dentro de un día

Sesiones de entrenamiento

GET/sessionsLista sesiones — query params: day_id, limit, offset
GET/sessions/{id}Detalle de sesión con todos sus set_records
POST/sessionsGuarda sesión completa — body: { day_id, date, sets: [...] }
POST/sessions/skipRegistra día salteado — body: { day_id, date }
DELETE/sessions/{id}Elimina sesión (solo del mismo día)

Historial por ejercicio

GET/exercises/{id}/historySets agrupados por sesión — query: limit (default 10)

Estadísticas

GET/stats/weeklyRacha actual + volumen últimos 7 días
GET/stats/summaryTotal sesiones, ejercicios, volumen histórico

Inteligencia Artificial

POST/ai/recommendRecomendación de peso/reps — llama a Groq o fallback local

Perfil

GET/profileDevuelve perfil + settings del usuario
PUT/profileActualiza perfil (peso, altura, edad, sexo, idioma, tema)

8. Flujo de Git

Ramas principales main → producción (protegida, requiere PR) develop → integración (base para features) Ramas de trabajo feature/ → nueva funcionalidad feature/auth-login fix/ → corrección de bug fix/session-duplicate chore/ → tareas técnicas chore/setup-ci-pipeline

Flujo de trabajo diario

PasoComandoDescripción
1git checkout developPartir siempre desde develop
2git pull origin developActualizar con últimos cambios
3git checkout -b feature/nombreCrear rama para la historia
4— (trabajar) —Commits pequeños y frecuentes
5git push origin feature/nombreSubir rama
6Crear Pull Request → developCI corre tests automáticamente
7Merge a developSi CI pasa
8PR develop → mainAl final del sprint, dispara deploy

Convención de commits (Conventional Commits)

feat: nueva funcionalidad fix: corrección de bug chore: setup, dependencias, config test: agregar o modificar tests refactor: cambio de código sin nueva funcionalidad docs: documentación Ejemplos: feat: add POST /sessions endpoint with set records fix: prevent duplicate session on double submit chore: configure GitHub Actions CI pipeline test: add unit tests for progression service

9. Épicas, Historias y Tareas

ÉPICA 1 Setup e Infraestructura Infra

Historia 1.1 — Configurar el repositorio base

Como developer quiero tener el repositorio estructurado para trabajar frontend y backend en un solo lugar.

  • Crear repositorio en GitHub con nombre gym-ai (privado o público)
  • Crear estructura de carpetas /frontend y /backend en la raíz
  • Crear .gitignore raíz que cubra Python, Node, Angular, variables de entorno y venv
  • Crear README.md con descripción del proyecto, stack y cómo correrlo localmente
  • Crear rama develop como rama base de trabajo (proteger main)
  • Configurar reglas de protección de rama en GitHub (require PR, require CI pass)

Historia 1.2 — Configurar el proyecto FastAPI

Como developer quiero tener el proyecto Python inicializado con estructura profesional.

  • Crear entorno virtual: python -m venv venv
  • Crear requirements.txt con: fastapi, uvicorn, sqlalchemy, pydantic, python-dotenv, psycopg2-binary, httpx, pytest, pytest-asyncio
  • Crear estructura de carpetas: app/routers, app/services, app/models, app/schemas, app/core, tests/
  • Crear app/main.py con FastAPI inicializado y CORS configurado
  • Crear app/core/config.py que lea variables desde .env
  • Crear app/core/database.py con SQLAlchemy engine y session factory
  • Crear .env.example con todas las variables necesarias (sin valores reales)
  • Verificar que GET /health responde 200 con {"status": "ok"}
  • Verificar que los docs de OpenAPI son accesibles en /docs

Historia 1.3 — Configurar el proyecto Angular

Como developer quiero el proyecto Angular inicializado con la configuración profesional.

  • Crear proyecto: ng new frontend --standalone --routing --style=scss
  • Habilitar strict: true en tsconfig.json
  • Instalar dependencias: @supabase/supabase-js, @ngrx/signals
  • Crear estructura de carpetas: core/, features/, shared/, environments/
  • Configurar environment.ts y environment.prod.ts con URL de Supabase y API
  • Configurar app.config.ts con provideHttpClient(withInterceptors([...]))
  • Crear app.routes.ts con lazy routes para cada feature
  • Verificar que ng serve y ng build corren sin errores

Historia 1.4 — Configurar Supabase

Como developer quiero tener la base de datos y autenticación configuradas.

  • Crear proyecto en Supabase (free tier)
  • Ejecutar SQL para crear tabla user_profiles con todas las columnas
  • Ejecutar SQL para crear tabla workout_days
  • Ejecutar SQL para crear tabla exercises
  • Ejecutar SQL para crear tabla sessions
  • Ejecutar SQL para crear tabla set_records
  • Activar Row Level Security en las 5 tablas
  • Crear políticas RLS: user_id = auth.uid() para SELECT, INSERT, UPDATE, DELETE
  • Habilitar auth por email/contraseña en Supabase Dashboard
  • Copiar SUPABASE_URL, SUPABASE_ANON_KEY y JWT_SECRET al .env del backend y environment.ts del frontend
  • Verificar conexión desde FastAPI con una query simple

Historia 1.5 — Configurar AWS EC2

Como developer quiero el servidor listo para deployar el backend.

  • Crear instancia EC2 t2.micro con Amazon Linux 2023 (free tier)
  • Generar y guardar par de claves SSH (.pem)
  • Configurar Security Group: puertos 22 (SSH), 80 (HTTP), 443 (HTTPS)
  • Conectarse por SSH: ssh -i key.pem ec2-user@[ip-publica]
  • Instalar Python 3.11, pip, git y nginx en la instancia
  • Configurar Nginx como reverse proxy: puerto 80 → uvicorn :8000
  • Crear archivo systemd para que FastAPI arranque como servicio permanente
  • Instalar Certbot y configurar HTTPS con Let's Encrypt (requiere dominio)
  • Verificar que GET /health responde desde la IP pública
  • Configurar variables de entorno en EC2 (archivo /etc/environment o .env del proyecto)

ÉPICA 2 Autenticación Must have

Historia 2.1 — Registro de usuario

Como usuario nuevo quiero poder registrarme para tener mi cuenta.

  • Crear AuthService en Angular que encapsula supabase.auth
  • Crear RegisterComponent con formulario reactivo (email, password, confirmar password)
  • Implementar validaciones: email válido, password mínimo 8 caracteres, confirmación coincide
  • Llamar a supabase.auth.signUp() y manejar errores (email ya existe, etc.)
  • Crear endpoint POST /auth/register en FastAPI que inserta fila en user_profiles
  • Llamar al endpoint del backend después del signup exitoso para crear el perfil
  • Mostrar mensaje "Revisá tu email para confirmar tu cuenta"
  • Redirigir a /login tras registro exitoso

Historia 2.2 — Login de usuario

Como usuario registrado quiero poder iniciar sesión.

  • Crear LoginComponent con formulario de email y contraseña
  • Llamar a supabase.auth.signInWithPassword()
  • Guardar sesión en AuthService como signal: currentUser = signal(null)
  • Escuchar supabase.auth.onAuthStateChange() para mantener sesión tras reload
  • Redirigir a /home tras login exitoso
  • Mostrar mensaje de error específico si las credenciales son incorrectas
  • Implementar link "¿Olvidaste tu contraseña?" que llama a supabase.auth.resetPasswordForEmail()

Historia 2.3 — Logout y protección de rutas

Como usuario quiero cerrar sesión y que las rutas privadas estén protegidas.

  • Implementar logout() en AuthService que llama a supabase.auth.signOut()
  • Limpiar estado global al hacer logout y redirigir a /login
  • Crear AuthGuard (CanActivateFn) que verifica sesión activa
  • Aplicar AuthGuard a todas las rutas excepto /login y /register
  • Crear AuthInterceptor que agrega Authorization: Bearer {jwt} a cada request HTTP
  • En FastAPI, crear async def get_current_user(token: str = Depends(oauth2_scheme))
  • Validar JWT usando SUPABASE_JWT_SECRET en cada endpoint protegido
  • Retornar 401 si el token es inválido o expirado

Como usuario quiero ver mis sesiones anteriores para revisar mi progreso.

  • Implementar GET /sessions?day_id={id} con join a set_records
  • Ordenar por fecha descendente, excluir sesiones skipped
  • Calcular volumen total por sesión en el backend (suma peso × reps)

  • Expandir sesión para ver el detalle de sets por ejercicio
  • Resaltar visualmente el mejor peso registrado por ejercicio

Historia 5.2 — Estadísticas semanales

Como usuario quiero ver mi racha y volumen semanal para mantener la motivación.

  • Implementar GET /stats/weekly en FastAPI
  • Calcular racha: días consecutivos hacia atrás con al menos una sesión no skipped
  • Calcular volumen: suma de (weight × reps) de todos los sets de los últimos 7 días
  • Mostrar racha y volumen como cards en el home dashboard
  • Refrescar estadísticas automáticamente después de guardar una sesión
  • Agregar tests unitarios para la lógica de cálculo de racha (edge cases: 0 días, racha larga)

Historia 5.3 — Historial por ejercicio

Como usuario quiero ver la evolución de un ejercicio a lo largo del tiempo.

  • Implementar GET /exercises/{id}/history — sets agrupados por sesión, últimas 10
  • Devolver por sesión: fecha, peso máximo, reps totales, volumen
  • Mostrar tendencia respecto a la sesión anterior (↑ subió / ↓ bajó / → igual)
  • Crear componente de historial accesible desde la tarjeta de ejercicio

ÉPICA 6 Integración IA (Groq) Should have

Historia 6.1 — Endpoint de recomendación en FastAPI

Como sistema quiero un endpoint seguro que llame a Groq sin exponer la API key.

  • Crear POST /ai/recommend con schema de request: exercise_id, exercise_name, unit, history
  • Construir prompt estructurado con historial del ejercicio y datos del usuario (perfil)
  • Llamar a Groq API con modelo llama-3.3-70b-versatile usando httpx async
  • Parsear respuesta JSON de Groq con manejo de errores robusto
  • Implementar fallback local si Groq falla: lógica de progresión basada en reglas
  • Retornar { sets: [{weight, reps}], reason, source: "groq"|"local" }
  • Agregar timeout de 5 segundos en la llamada a Groq
  • Agregar tests unitarios para el fallback local con diferentes escenarios de historial

Historia 6.2 — Cache de recomendaciones

Como sistema quiero cachear las recomendaciones para no llamar a Groq innecesariamente.

  • Implementar cache en memoria en FastAPI (dict) con clave exercise_id + last_session_date
  • Invalidar cache del ejercicio cuando se registra una nueva sesión que lo contiene
  • Agregar header X-Cache: HIT|MISS en la respuesta para debugging
  • Limitar cache a 200 entradas máximo (limpiar las más antiguas con LRU simple)

Historia 6.3 — Mostrar recomendaciones en el frontend

Como usuario quiero ver la sugerencia de IA para saber qué peso usar.

  • Llamar a POST /ai/recommend al abrir un ejercicio en modo entrenamiento
  • Mostrar skeleton loader mientras se obtiene la recomendación
  • Mostrar badge "IA" con peso y reps sugeridos junto al ejercicio
  • Pre-rellenar los inputs de la primera serie con los valores sugeridos
  • Permitir que el usuario edite los valores pre-rellenados sin restricciones
  • Mostrar razón de la recomendación en texto pequeño (colapsable)
  • Mostrar indicador "modo offline" si la recomendación viene del fallback local
  • No bloquear el entrenamiento si la llamada a IA falla — degradar gracefully

11. Guía de Deploy

Deploy manual inicial — Backend en EC2

# 1. Conectarse a EC2 ssh -i gym-ai-key.pem ec2-user@[ip-publica] # 2. Clonar repositorio git clone https://github.com/tuusuario/gym-ai.git cd gym-ai/backend # 3. Crear entorno virtual e instalar dependencias python3.11 -m venv venv source venv/bin/activate pip install -r requirements.txt # 4. Crear archivo .env con variables de producción nano .env # Pegar DATABASE_URL, SUPABASE_JWT_SECRET, GROQ_API_KEY # 5. Crear servicio systemd sudo nano /etc/systemd/system/gym-api.service
# /etc/systemd/system/gym-api.service [Unit] Description=GYM-AI FastAPI After=network.target [Service] User=ec2-user WorkingDirectory=/home/ec2-user/gym-ai/backend Environment="PATH=/home/ec2-user/gym-ai/backend/venv/bin" ExecStart=/home/ec2-user/gym-ai/backend/venv/bin/uvicorn app.main:app --host 0.0.0.0 --port 8000 Restart=always [Install] WantedBy=multi-user.target
# 6. Habilitar y arrancar el servicio sudo systemctl daemon-reload sudo systemctl enable gym-api sudo systemctl start gym-api sudo systemctl status gym-api # verificar que está running # 7. Configurar Nginx sudo nano /etc/nginx/conf.d/gym-api.conf
# /etc/nginx/conf.d/gym-api.conf server { listen 80; server_name [tu-dominio.com]; location / { proxy_pass http://127.0.0.1:8000; proxy_set_header Host $host; proxy_set_header X-Real-IP $remote_addr; proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for; } } # Habilitar Nginx y obtener certificado SSL sudo systemctl restart nginx sudo certbot --nginx -d tu-dominio.com

Reglas generales

  • Sin comentarios obvios — el código bien nombrado es auto-documentado
  • Funciones de máximo 30 líneas — si es más larga, dividir
  • Un archivo = una responsabilidad principal
  • Variables de entorno siempre en .env, nunca hardcodeadas
  • Nunca commitear .env, venv/, node_modules/, __pycache__/
  • Toda lógica de negocio en services/services — nunca en controllers/components
  • Tests nombrados: test_[qué hace]_[condición]_[resultado esperado]

13. Seguridad

Principio base: Las API keys de terceros (Groq) nunca tocan el cliente. Toda llamada a servicios externos se realiza desde el backend. El frontend solo maneja el JWT de Supabase.
